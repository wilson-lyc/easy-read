# EasyRead — 产品概要设计

## 1. 产品定位

**EasyRead** 是一款基于大语言模型（LLM）的 Obsidian 翻译插件，旨在帮助用户在笔记阅读与写作场景中无缝跨越语言障碍。核心价值在于**不打断工作流**——在 Obsidian 内完成选词、翻译、理解的全过程。支持在 **Markdown 笔记**和 **PDF 文档**中进行划词翻译。

### 1.1 核心场景

| 场景 | 描述 | 优先级 |
|------|------|--------|
| **划词翻译（Markdown）** | 阅读英文笔记时选中陌生词汇或段落，即时获得翻译 | P0 |
| **划词翻译（PDF）** | 在 Obsidian 中打开 PDF 文献，选中文字段落即时翻译 | P0 |
| **全文翻译** | 将整篇笔记从源语言翻译为目标语言，保留 Markdown 格式 | P0 |
| **双语对照** | 全文翻译后以原文+译文的对照方式展示 | P1 |
| **学术摘要翻译** | 快速翻译论文摘要或技术文档片段 | P1 |

### 1.2 目标用户

- **双语研究者/学生**：阅读英文论文（PDF）、技术文档，需要快速理解
- **外语笔记创作者**：在笔记中混合使用多种语言
- **Obsidian 深度用户**：希望保持"All in One"工作流，不想切到外部翻译工具

| F-02 | 划词翻译（PDF） | 在 Obsidian PDF 预览视图中选中文本，通过浮动按钮/右键触发翻译，气泡覆盖显示在 PDF 页面上方 | P0 |

| 需求 ID | 维度 | 描述 | 优先级 |
|---------|------|------|--------|
| N-01 | 性能 | 划词翻译应在 <2s 内返回（50词以内） | P0 |
| N-02 | 隐私 | 用户可选择本地 LLM（Ollama）以完全离线运行，数据不出设备 | P0 |
| N-03 | 可靠性 | 全文翻译中如果 API 中断，应有重试机制和断点续译 | P1 |
| N-04 | 可扩展性 | 插件架构应允许轻松添加新的 LLM 提供商 | P1 |
| N-05 | 占用 | 插件本身不应影响 Obsidian 启动速度（懒加载） | P1 |

## 3. 用户流程

### 3.1 划词翻译流程（Markdown）

```
1. 用户在 Markdown 编辑器中选中文本
2. 触发翻译（快捷键 Ctrl+T / 右键菜单 / 选中后出现的浮动按钮）
3. 插件获取选中文本，发送请求显示加载动画
4. LLM 返回翻译结果
5. 以 Tooltip 气泡形式展示在选中文本附近
6. 用户按 Esc 或点击外部关闭；或点击"替换原文"将译文写回编辑器
```

### 3.2 划词翻译流程（PDF）

```
1. 用户打开 Obsidian 内置 PDF 预览（PDF 嵌入模式或独立标签页）
2. 在 PDF 页面上选中一段文字
3. 浮动按钮出现在选中区域附近，或用户通过右键菜单触发
4. 插件读取 PDF 选中文本，处理连字符/换行等排版 artifact
5. 发送请求到 LLM，显示加载动画（Tooltip 内嵌 spinner）
6. LLM 返回翻译结果
7. 以浮动 Tooltip 覆盖在 PDF 页面上方展示译文
8. 用户按 Esc 关闭，或拖拽 Tooltip 避免遮挡 PDF 原文
```

### 3.3 全文翻译流程（Markdown）

```
1. 用户通过命令面板执行"EasyRead: 全文翻译"
2. 弹出设置弹窗：源语言（可选）/ 目标语言 / 输出方式（覆盖/新建/双语对照）
3. 用户确认后开始翻译
4. 插件分块发送文本（保留 Markdown 上下文）
5. 逐块接收翻译结果，流式输出到编辑器
6. 翻译完成后显示完成通知
```

## 4. 竞品分析

| 产品 | 优势 | 劣势 |
|------|------|------|
| **Google Translate (外部)** | 免费、支持语言多 | 需切换窗口，无 Obsidian 集成，不可定制 |
| **OpenAI Translator** | 基于 LLM，翻译质量高 | 独立应用，无 Obsidian 插件 |
| **Obsidian 内置插件** | 生态内原生 | 翻译质量一般，配置有限 |
| **Copilot (划词翻译)** | 体验轻量 | 非 Obsidian 专用，无全文翻译 |

**EasyRead 差异化优势**：原生 Obsidian 集成 + LLM 高质量翻译 + 多提供商可选（尤其是本地 LLM 保障隐私）+ 自定义翻译风格。

## 5. 功能边界与约束

- **PDF 页面层级**：PDF 划词翻译以页面为视觉容器，Tooltip 定位基于 PDF 视口坐标而非编辑器坐标
- **PDF 文本质量**：PDF 文本提取需处理连字符断词（hyphenation）、人工换行、特殊 Unicode 字符等
- **输入大小限制**：全文翻译时单次发送不超过 LLM Context Window 的 70%（预留 System Prompt 空间）
- **速率限制**：遵守各 API 提供商的 Rate Limit，队列化管理请求
- **经济性**：公开 Token 消耗估算，建议用户选择性价比高的模型（如 Claude 3.5 Sonnet / GPT-4o-mini）
- **网络依赖**：除 Ollama 外，其他提供商需要网络连接，插件应有网络状态提示

| 目标 | 关键结果 |
|------|---------|
| 翻译速度 | 划词翻译 P50 延迟 < 1.5s |

| 采用率 | 插件发布后 3 个月内 DAU > 500 |
| 稳定性 | 全文翻译成功率 > 98%（含重试） |
